  3. Type tProjectInfo
  4.   lTemplateVersion As Long              ' 0x000 (000d) VB compatible version
  5.   aObjectTable As Long                  ' 0x004 (004d) Pointer to ObjectTable
  6.   lNull1 As Long                        ' 0x008 (008d)
  7.   aStartOfCode As Long                  ' 0x00C (012d) Pointer to the start of some Assembly listing
  8.   aEndOfCode As Long                    ' 0x010 (016d) Pointer to the end of some Assembly listing
  9.   lDataBufferSize As Long               ' 0x014 (020d) Size of Data Buffer
  10.   aThreadSpace As Long                  ' 0x018 (024d)
  11.   aVBAExceptionhandler  As Long         ' 0x01C (028d) Pointer to VBA Exception Handler
  12.   aNativeCode As Long                   ' 0x020 (032d) Pointer to the start of RAW Native Code
  13.   uIncludeID(MAX_PATH * 2 + 7) As Byte  ' 0x024 (042d) VB Include Internal Project Identifier
  14.                                                        -> (260 + NullTerm) * 2 [unicode] + "*A"
  15.                                                        -> 528 bytes
  16.   aExternalTable As Long                ' 0x234 (564d) Pointer to API Imports Table
  17.   lExternalCount As Long                ' 0x238 (568d) Number of API's imported
  18.                                         ' 0x23C (572d) <-- Structure Size
  19. End Type

This gives the absolute maximum boundary of the structure. JoCo noted that he thought the structure was 0x1F4. Well, we can see that if you dont include the original path name (which, I think is null-terminated variable length) then it is merely 0x2A. If you do include the filename then it is of variable length, which is Bad Thing(tm) heh.

I've also done a few other tests to probe the length of this structure *including* the length of the filename... and so we come to what im thinking now. It seems that the size could be ludicrously large... 0x23C bytes (LoL). It seems to fit (MrU, try changing your size of that structure to colour in a 0x23C size block and you'll see it fits nicely).
